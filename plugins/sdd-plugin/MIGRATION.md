# Migration Guide — 0.x → v1.0.0

## What changed

v1.0.0 is a major architectural rewrite (Arch B: stage-as-subagent). Main session becomes a thin FSM dispatcher; each pipeline stage runs as a single sub-agent that inlines work + reviews + Skills.

**Headline numbers** (per design/00-architecture.md §5):
- Main session per-Issue token budget: ~19,715 → ~2,610 (~87% drop)
- Total system tokens: roughly flat (sub-agent context grows; main shrinks)
- Wall-clock per stage: slightly longer (~+60s per review round, serial reviewers)

See `design/00-architecture.md` for full architecture, `design/05-rethink-decisions.md` for the 10 RETHINK decisions, `design/SYNTHESIS-v2.md` for review-driven refinements.

## What is identical (zero migration steps needed for typical users)

- All `/sdd <command>` names and argument shapes
- All GitHub labels (`sdd:analyze`, `sdd:design`, `sdd:implement`, `sdd:test`, `sdd:done`, `sdd:child`, `sdd:review:deep`, `sdd:review:shallow`)
- All comment markers (`<!-- sdd:* -->`)
- `.github/.sdd-config` format and `skip-review:` key + 5 values (`analyze`, `design`, `implement`, `pr`, `qa`)
- `.github/.sdd-lang` format + language aliases (`en`, `ko`/`korean`/`한국어`, `ja`/`japanese`/`日本語`)
- Issue templates per language
- Stage outputs (markdown structure inside comments)
- Multilingual parent reference regex `(Parent|상위 |親)Issue: #<n>`
- Findings JSON schema
- Bash heuristic discipline + comment posting temp-file pattern

## Action required (typical user)

**None.** Update the plugin and continue using `/sdd` commands as before.

## In-flight Issue behavior

If you have Issues mid-pipeline at the time of upgrade:

| Stage label | v1.0.0 behavior |
|---|---|
| `sdd:analyze` (with output comment) | Continues from review/design stage normally |
| `sdd:design` | Continues from implement stage normally |
| `sdd:implement` (with branch + open PR) | Auto-routes to retry mode (NEW per R8) |
| `sdd:implement` (parent waiting for children) | Resumes once all children reach `sdd:done` |
| `sdd:test` | QA + close normally |
| `sdd:done` | No action |

### Edge case: TDD branches with partial 0.x commits
If you re-run `/sdd implement <N>` on a branch that has some TDD step commits from 0.x:
- v1.0.0 will check for prior step completion via the test-evidence marker (R9)
- 0.x branches may not have all expected markers → some steps may re-execute
- **Recommendation**: finish the in-flight Issue on 0.x before upgrading, or restart fresh after upgrading

## Mixed-version teams warning

If your team has some members on 0.x and others on v1.0.0, do not have them resume each other's Issues mid-pipeline. The R9 idempotency markers added by v1.0.0 are ignored by 0.x; the R8 auto-route in v1.0.0 may surprise 0.x users.

**Recommendation**: upgrade your team together.

## Behavior changes (small, but worth knowing)

1. **`/sdd implement <N>` after a PR exists**: 0.x errored ("PR already exists"). v1.0.0 auto-routes to retry mode. If you have CI scripts that parsed the old error, update them.

2. **TDD step idempotency**: re-running implement after a partial completion skips already-done steps (faster resume).

3. **`/sdd init` partial failure**: 0.x left half-created labels. v1.0.0 rolls back successful creates on any failure. If the rollback itself fails, the error message includes manual cleanup commands.

4. **Per-reviewer model dial**: in Arch B, individual reviewers inside a stage sub-agent run sequentially in the same context. The depth label still affects:
   - `/code-review` effort (default → high; deep → max; shallow → medium)
   - `/security-review` skipped on shallow
   - Overall reasoning quality (stage sub-agent's chosen model)
   
   Per-reviewer model assignment (e.g. completeness=sonnet, adversarial=opus) is now informational, not enforced.

5. **Wall-clock per stage**: slightly longer than 0.x because reviews run serially inside the stage sub-agent (no parallel Agent calls). Trade-off: main session context drops by ~65%. For `/sdd auto` runs, this is a net win.

## R7: `commands/ai-review-*.md` → `commands/atoms/rubrics/*.md`

The 14 review rubric files moved location and lost their `ai-review-` prefix. Example:
- 0.x: `commands/ai-review-analyze-completeness.md`
- v1.0.0: `commands/atoms/rubrics/analyze-completeness.md`

No user impact unless you forked/customized the rubric files or have automation referencing the old paths.

## CI/CD: `/sdd batch` regeneration

If you have a saved `.github/.sdd-batch.sh` from 0.x and your CI re-uses it, regenerate it under v1.0.0 to pick up the R8/R9 improvements. The script template carries a version watermark (`# generated by sdd-plugin <version>`) to make this visible.

## Rollback policy

If v1.0.0 ships with a critical regression:

- **Option A — fresh installs**: marketplace re-pinned to v0.37.0 (the last 0.x sync release). Users who already auto-pulled v1.0.0 are not auto-downgraded.
- **Option B — hotfix v1.0.1** (preferred): patch released forward. R9 markers left in user repos are no-op for 0.x and harmless.

### Downgrade caveat
v1.0.0 may have added `<!-- sdd:tdd:step-N -->` style markers to commit bodies. These persist after downgrade and look unfamiliar to 0.x but cause no functional problems.

## Version-sync rule (developer-facing)

The `plugin.json` and `marketplace.json` versions must move together. A CI check (`.github/workflows/version-sync.yml`) enforces this on every PR.

## Support

File regressions at https://github.com/dev-yakuza/deku-claude-plugins/issues with the label `sdd-v1.0.0`.

## Known limitations / deferred

- `/sdd review --deep` (adversarial inclusion) deferred to v1.1 — file an issue if you need it sooner.
- `/sdd retroactive-mark` not needed thanks to sha-based R9 idempotency.
- Sandbox toggle UX (~190 lines in `auto.md` Phase 3.1 step 5) preserved verbatim; UX simplification deferred to v1.1+.
